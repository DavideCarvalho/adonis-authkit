AuthKit ships an opt-in **admin console**: a metrics dashboard, user/role management,
session context, organization management, the OAuth client list, the audit log, an
impersonation panel, and a full runtime settings editor. It is **off by default** â€”
existing hosts don't change behaviour.

<Callout type="info">
  **The console is now a React SPA by default.** Starting in the current release the admin
  console is a self-contained, Vite-built React application (hash-routing, dark/light theme,
  violet accent `#625fff`) bundled **inside** the npm package â€” zero extra build step for the
  host. The classic Edge server-rendered console is still available as an opt-in alternative
  (`ui: 'edge'`).
</Callout>

## Enabling it

The console must be turned on in **both** places â€” `defineConfig` (the guard) and
`registerAuthHost` (route mounting happens before the lazy config resolves):

```ts title="config/authkit_server.ts"
defineConfig({
  // ...
  admin: {
    enabled: true,
    roles: ['ADMIN'], // global roles allowed into /admin; default ['ADMIN']
  },
})
```

```ts title="start/routes.ts"
registerAuthHost(router, {
  mountPath: '/oidc',
  admin: true, // console under /admin, React SPA (default)
})
```

| Field | Type | Default | Notes |
| --- | --- | --- | --- |
| `enabled` | `boolean` | `false` | Mounts the admin console group behind the admin guard. |
| `roles` | `string[]` | `['ADMIN']` | Global roles that grant access to the console. |

### Configurable admin prefix

By default the console is mounted under `/admin`. You can move it to any path by passing
`{ prefix: '...' }` instead of `true` in `registerAuthHost`:

```ts title="start/routes.ts"
// Default â€” console under /admin (backward-compatible)
registerAuthHost(router, {
  mountPath: '/oidc',
  admin: true,
})

// Custom prefix â€” console under /auth/admin
registerAuthHost(router, {
  mountPath: '/oidc',
  admin: { prefix: '/auth/admin' },
})
```

The prefix is normalised automatically: it always starts with `/` and never has a trailing
slash. All console routes, API calls, and assets follow the effective prefix â€” no further
configuration is required. The Admin REST API (`/api/authkit/v1`) is **not** affected by
this option.

**Acme example** â€” Acme mounts the console at `/auth/admin` to avoid collisions with an
existing `/admin` route in their main app:

```ts title="start/routes.ts"
registerAuthHost(router, {
  mountPath: '/oidc',
  admin: { prefix: '/auth/admin' },
  adminApi: true,
})
```

The dashboard is then at `https://auth.acme.com/auth/admin`. The SPA uses hash-routing
(`#/users`, `#/settings`, â€¦), so all page URLs are served by the same shell at `{prefix}`.

<Callout type="info">
  When `prefix` is omitted inside the object form (`admin: {}`), the default `/admin` is
  used â€” identical to `admin: true`.
</Callout>

### UI mode: React SPA (default) vs Edge (legacy)

`registerAuthHost` accepts an optional `ui` field alongside `prefix`:

```ts title="start/routes.ts"
// React SPA â€” default; zero extra setup
registerAuthHost(router, {
  mountPath: '/oidc',
  admin: true,              // equivalent to { ui: 'react' }
})

// Classic Edge server-rendered console (back-compat)
registerAuthHost(router, {
  mountPath: '/oidc',
  admin: { ui: 'edge' },
})

// React SPA with a custom prefix
registerAuthHost(router, {
  mountPath: '/oidc',
  admin: { prefix: '/auth/admin', ui: 'react' },
})
```

| `ui` value | Behaviour |
| --- | --- |
| `'react'` (default) | Vite-built SPA shipped inside the package â€” no build step needed. Hash-routing. JSON API under `{prefix}/api/*`. Static assets under `{prefix}/assets/*` (aggressive cache). |
| `'edge'` | Classic server-rendered Edge views. Same feature set; all form actions are traditional `POST` submissions. Full back-compat with previous releases. |

#### React SPA internals

The SPA (`ui: 'react'`) is:

- **Pre-built and bundled inside `@dudousxd/adonis-authkit-server`** â€” the Vite output lands
  in `build/host/ui-dist/` at publish time and is served directly by `AdminShellController`.
  No separate build, no Vite dev server, no extra npm scripts on the host.
- **Hash-routing** â€” every URL like `{prefix}#/users` or `{prefix}#/settings` is served by
  the same `index.html` shell. Deep links work without server-side route matching.
- **JSON API + TanStack Query** â€” the SPA fetches all data client-side via
  `@dudousxd/adonis-authkit-react`'s typed client and TanStack Query hooks
  (`useUsersQueryOptions`, `useCreateUserMutationOptions`, etc.). The API endpoints live
  under `{prefix}/api/*` and are protected by the same `adminGuard` as the shell.
  See **[Typed Client & TanStack Query](/docs/data-fetching)** for the full hook catalog â€”
  the SPA is the canonical dogfood implementation; if you are building a custom admin
  interface, use the same hooks.
- **`AuthkitClientProvider`** â€” the shell wraps the app in `<QueryClientProvider>` and
  `<AuthkitClientProvider>`. The client reads `window.__AUTHKIT__.endpoints.api` for the
  base URL and `window.__AUTHKIT__.csrfToken` for CSRF â€” no hard-coded paths anywhere.
- **Assets** â€” `{prefix}/assets/:file` serves the hashed JS/CSS files with
  `Cache-Control: public, max-age=31536000, immutable`.
- **Dark/light theme** â€” respects the OS preference (`prefers-color-scheme`). Accent colour
  `#625fff`.
- **`window.__AUTHKIT__` config** â€” the shell injects a config object at render time
  (`adminBase`, `csrfToken`, `currentUser`, `endpoints.api`). The SPA never hard-codes paths.

If the built dist is absent (a dev environment before running the server build), the shell
falls back to a minimal HTML page with build instructions.

## The guard

The `adminGuard` (also exported) protects the group. It requires:

1. `admin.enabled: true` â€” when the console is **disabled**, the guard responds with **404
   Not Found** (the routes simply don't exist as far as a caller can tell);
2. an active **account session** (otherwise redirect to `/account/login`);
3. the signed-in account having at least one of `admin.roles` in its global roles
   (otherwise redirect to `/account/tokens`, so the existence of `/admin` isn't leaked).

The allowed roles are resolved at runtime from the `authkit.server` config. See
[Security](/docs/security) for how the disabled-console 404 fits the broader hardening.

## Dashboard

`GET /admin` shows a metrics overview built from the audit sink:

- **Total users** â€” count from the account store.
- **Active sessions** â€” count of current sessions (when the adapter supports enumeration).
- **MAU (Monthly Active Users)** â€” unique accounts with a `login.success` in the last 30 days.
- **Daily sign-ins / sign-ups** â€” server-side SVG line charts for the last 30 days.

When the audit sink does not support `list`, the charts are omitted and only the static
counts are shown.

## Session context

Every session displayed in the console (user detail page) carries rich context captured
at login time:

| Field | Source |
| --- | --- |
| Browser | Parsed from the User-Agent string (e.g. `Chrome 124`) |
| OS | Parsed from the User-Agent string (e.g. `macOS`) |
| IP address | Request IP at login |
| Location | Resolved via the host's `resolveGeo` hook (optional) |

To add geolocation, plug in a `resolveGeo` hook in `defineConfig`:

```ts title="config/authkit.ts"
resolveGeo: async (ip) => {
  // example: return 'SĂŁo Paulo, BR'
  const geo = await myGeoLookup(ip)
  return geo ? `${geo.city}, ${geo.countryCode}` : null
},
```

The lookup is **fail-safe with a 1.5 s timeout**: an error or timeout returns `null` (no
location shown) and never delays the listing.

## Sessions & revocation

Each user row links to **`/admin/users/:id/sessions`**, which lists the account's active
**sessions** (IdP logins) and **grants** (per-client authorizations), with a live count of
access / refresh tokens per grant. A single **"Revoke all sessions and grants"** button
destroys them via `AdminSessionsService`.

Revocation works by destroying the account's `Session` and `Grant` artifacts directly
through the OIDC adapter. Destroying a grant **cascades** to its tokens: oidc-provider
validates every access/refresh token against `Grant.find(token.grantId)` and rejects it
with `invalid_token` when the grant is gone. As a belt-and-braces measure the service also
destroys the access/refresh token rows that reference the revoked grants (when the adapter
can enumerate them). The operation emits a `session.revoked_all` audit event with the
counts in `metadata`.

Like the client list, enumeration relies on the adapter's **optional** `list?()` capability
(the database and Redis adapters both implement it). When the adapter can't enumerate, the
page **degrades gracefully** with a notice instead of erroring. Both the database adapter's
table query and the Redis adapter's `SCAN` are scoped per model, so the lookup is cheap.

## Impersonation panel

When the `admin_impersonation` runtime setting is enabled (`{ "enabled": true }`), a user's
detail page shows an **Impersonation** panel with the exact `token_exchange` parameters
needed to assume that user's identity (RFC 8693). This surfaces the `subject_token_type`,
`requested_token_type`, and a ready-to-run `curl` snippet â€” it does **not** perform the
exchange itself.

```bash
# Enable via runtime setting (no redeploy)
node ace authkit:settings:set admin_impersonation '{"enabled":true}'
```

<Callout type="warn">
  The exchange requires a confidential client with the `urn:ietf:params:oauth:grant-type:token-exchange`
  grant and a valid admin access token as `subject_token`. The panel shows the parameters;
  your client performs the actual request. Viewing the panel emits an
  `impersonation.started` audit event.
</Callout>

## User management (create / invite / reset / disable / delete)

The users page has a **create** form (email + optional name + optional password). When you
leave the password blank the account is created with a throwaway password and a
**password-reset email** is sent â€” the user sets their own password (the "invite" flow).
Each row also exposes **Send password reset** and, when the store supports it,
**Disable** / **Enable** buttons. The operations emit `user.created`, `user.password_reset_sent`, `user.disabled`,
`user.enabled`, and `user.deleted` audit events. The **Delete** action runs the same cascade
as the self-service flow: sessions, grants, PATs, passkeys, provider identities, and org
memberships are removed; audit records are anonymized. See [Compliance](/docs/compliance)
for the full cascade details.

Disabling requires the optional **`AccountStatusCapability`** on the account store (see
[Account store](/docs/account-store)). The Lucid store implements it via a
`disabled_at` timestamp column **only when the column exists on your model** â€” otherwise the
capability is genuinely absent and the disable/enable buttons are hidden. Add the column with
a migration of your own:

```ts
table.timestamp('disabled_at').nullable()
```

Both login paths (the OIDC interaction and the account-console login) reject disabled
accounts with an "account disabled" message, even with a correct password.

## Client CRUD

All clients are managed at runtime â€” there is no static client list in config. Create them
via this console, the [Admin REST API](/docs/admin-api#clients), or the
`authkit:clients:create` command (see [Reference](/docs/reference#authkitclientscreate)).
No redeploy needed.

The clients page lists all **adapter-stored** clients created via the
console or [Dynamic Registration](/docs/dynamic-registration). Adapter-stored clients are
fully manageable via `AdminClientsService`:

- **Create** â€” generates a `client_id` when omitted; for confidential clients
  (`tokenEndpointAuthMethod !== 'none'`) it generates a `client_secret` and shows it **once**
  (it isn't recoverable afterwards). Public clients get no secret.
- **Edit** â€” updates the editable metadata (redirect URIs, post-logout redirect URIs,
  grants, token-endpoint auth method, backchannel logout URI, `session_required`) while
  **preserving** the existing secret.
- **Regenerate secret** â€” issues a fresh `client_secret` for a confidential client (shown
  once); rejected for public clients.
- **Delete** â€” removes the client from the adapter.

Every write builds the payload in the exact snake_case shape the underlying oidc-provider
persists (matching what RFC 7591 dynamic registration writes) and then **evicts the
provider's dynamic-client cache** so the change takes effect immediately. Each operation
emits an audit event: `client.created`, `client.updated`, `client.deleted`.

Enumeration relies on the adapter's **optional** `list?()` capability (the legacy
`listClients?()` is still honored for back-compat and now delegates to `list()`). When the
adapter implements neither, the service's `canList` is `false` and the page **degrades
gracefully** â€” the create form still works, but the adapter-stored list is hidden with a
notice rather than erroring.

## Settings page

`GET {prefix}/settings` (Edge mode) or `{prefix}#/settings` (React SPA) shows a **Runtime
Settings** panel with multiple sections covering all 21 known setting keys. The page is the
primary interface for operational configuration that does not require a redeploy. See
[Runtime Settings](/docs/settings) for the full setting catalog, precedence rules, and
migration guide.

Every card shows a **badge** indicating whether the effective value comes from the runtime
setting (`from setting`) or falls back to the lib default (`from default`). When the
`auth_settings` table is absent, a notice appears with the `CREATE TABLE` snippet and submit
buttons are disabled.

Every save emits a `settings.updated` audit event.

#### Bot-protection card

The card shows the current effective state and lets an admin:

- **Enable / disable** bot protection for the whole IdP without a redeploy.
- **Override the protected flows** (`login`, `signup`, `reset`) independently of the
  config value.

**States:**

- **Config absent (`botProtection.verify` not set)** â€” the card is rendered in a disabled
  state with a notice that bot protection has not been wired up in `config/authkit.ts`. The
  form controls are non-interactive.
- **`auth_settings` table absent** â€” a notice explains that the `auth_settings` table is
  required for runtime toggles and shows the `CREATE TABLE` snippet. The form still
  renders so the admin can see the static-config values, but submit buttons are disabled.
- **Table present** â€” the form is live. Submitting it writes (or deletes) the
  `bot_protection` row and every subsequent request picks up the change within 15 s (the
  TTL of the in-process cache). A **Reset to config** button deletes the row and returns
  control to the static config.

See [Security â€” Runtime toggle](/docs/security#runtime-toggle) for the full precedence
rules and setting shape.

#### Registration card

Controls whether public sign-up is open. The static fallback is `config.registration.enabled`
(default `true`). Disabling closes the sign-up page and rejects new registrations; org
invitations and admin-created accounts are unaffected. A **Reset to config** button
removes the runtime setting and returns to the static default.

See [Security â€” Registration toggle](/docs/security#registration-toggle) for the full
behaviour and escape hatch.

#### Require verified email card

Overrides `config.login.requireVerifiedEmail` at runtime for all three login flows. Includes
an optional **grace period** field (`graceDays`) that allows newly-registered accounts to
log in for a configurable number of days without a verified email. When the `accountStore`
lacks the `isEmailVerified` capability the card shows a notice that the guard is a no-op.

See [Compliance â€” Email verification gate](/docs/compliance#verified-email-grace) for the
grace-period behaviour and [Security â€” Require verified email toggle](/docs/security#require-verified-email-toggle)
for the precedence rules.

#### Maintenance mode card

Activates a maintenance screen that blocks login / sign-up / forgot-password for
non-admin users. An optional custom message is shown on the maintenance page; leave it
blank to use the built-in i18n string. Admin accounts (roles matching `admin.roles`)
bypass the gate.

A **Reset** button removes the setting and takes the IdP out of maintenance mode. If the
console is unreachable during an incident, the Admin REST API escape hatch lets you disable
maintenance without a session:

```bash
curl -X PUT https://idp.acme.com/api/authkit/v1/settings/maintenance_mode \
  -H "Authorization: Bearer $KEY" \
  -H "Content-Type: application/json" \
  -d '{ "value": { "enabled": false } }'
```

See [Security â€” Maintenance mode](/docs/security#maintenance-mode) for the full behaviour.

#### Authentication methods card

Controls which login methods the sign-in screen offers at runtime â€” without a redeploy.

The card lists checkboxes for each method with contextual dependency hints:

- **Password** â€” the standard email + password form.
- **Magic link** â€” "send me a login link"; shown only when the mail capability and store
  support it. The checkbox is disabled with a hint when the capability is absent.
- **Passkey** â€” passkey-first sign-in; shown only when WebAuthn is configured.
- **Forgot password** â€” the "forgot my password" link. The checkbox is automatically
  disabled (and unchecked) when **Password** is off â€” there is nothing to reset without a
  password.
- **Social providers** â€” one checkbox per configured provider (e.g. Google, GitHub). Only
  providers present in the static config are listed; the setting cannot activate a provider
  the code does not have wired up.

A **Reset to config** button deletes the row and returns to the config-derived defaults.

See [Security â€” Authentication methods](/docs/security#authentication-methods) for the full
shape, precedence rules, and fail-safe behaviour.
