title: Admin Console
description: The opt-in IdP admin console — dashboard, users, sessions, organizations, clients, audit, and impersonation.
---

AuthKit ships an opt-in **admin console** mounted under `/admin/*`: a metrics dashboard,
user/role management, session context, organization management, the OAuth client list, the
audit log, and an impersonation panel. It is **off by default** — existing hosts don't
change behaviour.

## Enabling it

The console must be turned on in **both** places — `defineConfig` (the guard) and
`registerAuthHost` (route mounting happens before the lazy config resolves):

```ts title="config/authkit_server.ts"
defineConfig({
  // ...
  admin: {
    enabled: true,
    roles: ['ADMIN'], // global roles allowed into /admin; default ['ADMIN']
  },
})
```

```ts title="start/routes.ts"
registerAuthHost(router, {
  mountPath: '/oidc',
  admin: true, // mirror admin.enabled here
})
```

| Field | Type | Default | Notes |
| --- | --- | --- | --- |
| `enabled` | `boolean` | `false` | Mounts the `/admin/*` group behind the admin guard. |
| `roles` | `string[]` | `['ADMIN']` | Global roles that grant access to the console. |
| `impersonation` | `boolean` | `false` | Show the impersonation panel on user detail pages. |

## The guard

The `adminGuard` (also exported) protects the group. It requires:

1. `admin.enabled: true` — when the console is **disabled**, the guard responds with **404
   Not Found** (the routes simply don't exist as far as a caller can tell);
2. an active **account session** (otherwise redirect to `/account/login`);
3. the signed-in account having at least one of `admin.roles` in its global roles
   (otherwise redirect to `/account/tokens`, so the existence of `/admin` isn't leaked).

The allowed roles are resolved at runtime from the `authkit.server` config. See
[Security](/docs/security) for how the disabled-console 404 fits the broader hardening.

## Routes

| Route | Action |
| --- | --- |
| `GET /admin` | Dashboard — MAU, daily sign-ins, daily sign-ups (SVG charts), total users |
| `GET /admin/users` | List/search users |
| `POST /admin/users` | Create a user (with a password, or invite via password-reset email) |
| `POST /admin/users/:id/roles` | Replace a user's global roles (`AccountStore.setGlobalRoles`) |
| `POST /admin/users/:id/reset-password` | Issue a reset token and email it to the user |
| `POST /admin/users/:id/disable` | Disable the account (blocks login) — needs `AccountStatusCapability` |
| `POST /admin/users/:id/enable` | Re-enable a disabled account |
| `POST /admin/users/:id/delete` | Delete a user and cascade (LGPD/GDPR) |
| `GET /admin/users/:id/sessions` | List the account's active sessions + grants (with browser/OS/IP/location) |
| `POST /admin/users/:id/revoke-sessions` | Revoke all of the account's sessions + grants |
| `GET /admin/clients` | List OAuth clients (static config + adapter-stored) |
| `POST /admin/clients` | Create a dynamic client |
| `POST /admin/clients/:id/edit` | Edit a dynamic client |
| `POST /admin/clients/:id/regenerate-secret` | Regenerate a confidential client's secret |
| `POST /admin/clients/:id/delete` | Delete a dynamic client |
| `GET /admin/audit` | Browse the audit log (when the sink supports `list`) |
| `GET /admin/orgs` | List organizations (when tables exist) |
| `POST /admin/orgs` | Create an organization |
| `GET /admin/orgs/:id` | View org detail: members + pending invitations |
| `POST /admin/orgs/:id/delete` | Delete an organization and cascade |
| `POST /admin/orgs/:id/members` | Add a member |
| `POST /admin/orgs/:id/members/:accountId/remove` | Remove a member |
| `POST /admin/orgs/:id/invitations/:invId/revoke` | Revoke an invitation |

The console reads through the same contracts as the rest of the host kit:
`AccountStore.listAccounts` / `setGlobalRoles` for users, the OIDC adapter + static
`clients` for the client list, and `AuditSink.list` for audit. A write-only audit sink
degrades gracefully — the audit page shows "query not supported".

## Dashboard

`GET /admin` shows a metrics overview built from the audit sink:

- **Total users** — count from the account store.
- **Active sessions** — count of current sessions (when the adapter supports enumeration).
- **MAU (Monthly Active Users)** — unique accounts with a `login.success` in the last 30 days.
- **Daily sign-ins / sign-ups** — server-side SVG line charts for the last 30 days.

When the audit sink does not support `list`, the charts are omitted and only the static
counts are shown.

## Session context

Every session displayed in the console (user detail page) carries rich context captured
at login time:

| Field | Source |
| --- | --- |
| Browser | Parsed from the User-Agent string (e.g. `Chrome 124`) |
| OS | Parsed from the User-Agent string (e.g. `macOS`) |
| IP address | Request IP at login |
| Location | Resolved via the host's `resolveGeo` hook (optional) |

To add geolocation, plug in a `resolveGeo` hook in `defineConfig`:

```ts title="config/authkit.ts"
resolveGeo: async (ip) => {
  // example: return 'São Paulo, BR'
  const geo = await myGeoLookup(ip)
  return geo ? `${geo.city}, ${geo.countryCode}` : null
},
```

The lookup is **fail-safe with a 1.5 s timeout**: an error or timeout returns `null` (no
location shown) and never delays the listing.

## Sessions & revocation

Each user row links to **`/admin/users/:id/sessions`**, which lists the account's active
**sessions** (IdP logins) and **grants** (per-client authorizations), with a live count of
access / refresh tokens per grant. A single **"Revoke all sessions and grants"** button
destroys them via `AdminSessionsService`.

Revocation works by destroying the account's `Session` and `Grant` artifacts directly
through the OIDC adapter. Destroying a grant **cascades** to its tokens: oidc-provider
validates every access/refresh token against `Grant.find(token.grantId)` and rejects it
with `invalid_token` when the grant is gone. As a belt-and-braces measure the service also
destroys the access/refresh token rows that reference the revoked grants (when the adapter
can enumerate them). The operation emits a `session.revoked_all` audit event with the
counts in `metadata`.

Like the client list, enumeration relies on the adapter's **optional** `list?()` capability
(the database and Redis adapters both implement it). When the adapter can't enumerate, the
page **degrades gracefully** with a notice instead of erroring. Both the database adapter's
table query and the Redis adapter's `SCAN` are scoped per model, so the lookup is cheap.

## Impersonation panel

When `admin.impersonation: true`, a user's detail page shows an **Impersonation** panel
with the exact `token_exchange` parameters needed to assume that user's identity (RFC 8693).
This surfaces the `subject_token_type`, `requested_token_type`, and a ready-to-run
`curl` snippet — it does **not** perform the exchange itself.

```ts title="config/authkit.ts"
admin: {
  enabled: true,
  roles: ['ADMIN'],
  impersonation: true, // default: false
},
```

<Callout type="warn">
  The exchange requires a confidential client with the `urn:ietf:params:oauth:grant-type:token-exchange`
  grant and a valid admin access token as `subject_token`. The panel shows the parameters;
  your client performs the actual request. Viewing the panel emits an
  `impersonation.started` audit event.
</Callout>

## User management (create / invite / reset / disable / delete)

The users page has a **create** form (email + optional name + optional password). When you
leave the password blank the account is created with a throwaway password and a
**password-reset email** is sent — the user sets their own password (the "invite" flow).
Each row also exposes **Send password reset** and, when the store supports it,
**Disable** / **Enable** buttons. The operations emit `user.created`, `user.password_reset_sent`, `user.disabled`,
`user.enabled`, and `user.deleted` audit events. The **Delete** action runs the same cascade
as the self-service flow: sessions, grants, PATs, passkeys, provider identities, and org
memberships are removed; audit records are anonymized. See [Compliance](/docs/compliance)
for the full cascade details.

Disabling requires the optional **`AccountStatusCapability`** on the account store (see
[Account store](/docs/account-store)). The Lucid store implements it via a
`disabled_at` timestamp column **only when the column exists on your model** — otherwise the
capability is genuinely absent and the disable/enable buttons are hidden. Add the column with
a migration of your own:

```ts
table.timestamp('disabled_at').nullable()
```

Both login paths (the OIDC interaction and the account-console login) reject disabled
accounts with an "account disabled" message, even with a correct password.

## Client CRUD

<Callout type="warn">
  **Static clients are deprecated.** The recommended workflow is to create and manage all
  clients through this console (or the [Admin REST API](/docs/admin-api#clients)) — no config
  change or redeploy needed. When your `config/authkit.ts` still has a `clients: [...]` block,
  a **deprecation banner** appears at the top of this page reminding you to migrate. Run
  `node ace authkit:clients:import` to copy the static clients to the adapter/DB (see
  [Reference — authkit:clients:import](/docs/reference#authkitclientsimport)).
</Callout>

The clients page lists **both** the statically-configured `clients` (read-only, shown with
a "static — managed in config" label) and the **adapter-stored** ones created via the
console or [Dynamic Registration](/docs/dynamic-registration). Adapter-stored clients are
fully manageable via `AdminClientsService`:

- **Create** — generates a `client_id` when omitted; for confidential clients
  (`tokenEndpointAuthMethod !== 'none'`) it generates a `client_secret` and shows it **once**
  (it isn't recoverable afterwards). Public clients get no secret.
- **Edit** — updates the editable metadata (redirect URIs, post-logout redirect URIs,
  grants, token-endpoint auth method, backchannel logout URI, `session_required`) while
  **preserving** the existing secret.
- **Regenerate secret** — issues a fresh `client_secret` for a confidential client (shown
  once); rejected for public clients.
- **Delete** — removes the client from the adapter.

Every write builds the payload in the exact snake_case shape the underlying oidc-provider
persists (matching what RFC 7591 dynamic registration writes) and then **evicts the
provider's dynamic-client cache** so the change takes effect immediately. Each operation
emits an audit event: `client.created`, `client.updated`, `client.deleted`.

Enumeration relies on the adapter's **optional** `list?()` capability (the legacy
`listClients?()` is still honored for back-compat and now delegates to `list()`). When the
adapter implements neither, the service's `canList` is `false` and the page **degrades
gracefully** — static clients still render and the create form still works, but the
adapter-stored list is hidden with a notice rather than erroring. Statically-configured
clients are always read-only (they live in your config, not the adapter).

When the three organization tables are present, an **Organizations** tab appears at

## Settings page

`GET /admin/settings` shows a **Runtime Settings** panel with four cards: bot protection,
registration, require verified email, and maintenance mode.

| Route | Action |
| --- | --- |
| `GET /admin/settings` | Render the settings page. |
| `POST /admin/settings/bot-protection` | Save (upsert) the `bot_protection` setting. |
| `POST /admin/settings/bot-protection/reset` | Delete the setting — reverts to static config. |
| `POST /admin/settings/registration` | Save (upsert) the `registration` setting. |
| `POST /admin/settings/registration/reset` | Delete the setting — reverts to static config. |
| `POST /admin/settings/require-verified-email` | Save (upsert) the `require_verified_email` setting. |
| `POST /admin/settings/require-verified-email/reset` | Delete the setting — reverts to static config. |
| `POST /admin/settings/maintenance-mode` | Save (upsert) the `maintenance_mode` setting. |
| `POST /admin/settings/maintenance-mode/reset` | Delete the setting — disables maintenance mode. |

All four cards share the same state model: each shows a **badge** indicating whether the
effective value comes from the runtime setting (`from setting`) or falls back to the static
config (`from config`). When the `auth_settings` table is absent, a notice appears on every
card with the `CREATE TABLE` snippet and the submit buttons are disabled.

### Bot-protection card

The card shows the current effective state and lets an admin:

- **Enable / disable** bot protection for the whole IdP without a redeploy.
- **Override the protected flows** (`login`, `signup`, `reset`) independently of the
  config value.

**States:**

- **Config absent (`botProtection.verify` not set)** — the card is rendered in a disabled
  state with a notice that bot protection has not been wired up in `config/authkit.ts`. The
  form controls are non-interactive.
- **`auth_settings` table absent** — a notice explains that the `auth_settings` table is
  required for runtime toggles and shows the `CREATE TABLE` snippet. The form still
  renders so the admin can see the static-config values, but submit buttons are disabled.
- **Table present** — the form is live. Submitting it writes (or deletes) the
  `bot_protection` row and every subsequent request picks up the change within 15 s (the
  TTL of the in-process cache). A **Reset to config** button deletes the row and returns
  control to the static config.

Every write emits a `settings.updated` audit event with `metadata.key = "bot_protection"`.

See [Security — Runtime toggle](/docs/security#runtime-toggle) for the full precedence
rules and setting shape.

### Registration card

Controls whether public sign-up is open. The static fallback is `config.registration.enabled`
(default `true`). Disabling closes the sign-up page and rejects new registrations; org
invitations and admin-created accounts are unaffected. A **Reset to config** button
removes the runtime setting and returns to the static default.

See [Security — Registration toggle](/docs/security#registration-toggle) for the full
behaviour and escape hatch.

### Require verified email card

Overrides `config.login.requireVerifiedEmail` at runtime for all three login flows. When
the `accountStore` lacks the `isEmailVerified` capability the card shows a notice that the
guard is a no-op. A **Reset to config** button removes the runtime override.

See [Security — Require verified email toggle](/docs/security#require-verified-email-toggle)
for the full precedence rules.

### Maintenance mode card

Activates a maintenance screen that blocks login / sign-up / forgot-password for
non-admin users. An optional custom message is shown on the maintenance page; leave it
blank to use the built-in i18n string. Admin accounts (roles matching `admin.roles`)
bypass the gate.

A **Reset** button removes the setting and takes the IdP out of maintenance mode. If the
console is unreachable during an incident, the Admin REST API escape hatch lets you disable
maintenance without a session:

```bash
curl -X PUT https://idp.acme.com/api/authkit/v1/settings/maintenance_mode \
  -H "Authorization: Bearer $KEY" \
  -H "Content-Type: application/json" \
  -d '{ "value": { "enabled": false } }'
```

See [Security — Maintenance mode](/docs/security#maintenance-mode) for the full behaviour.
