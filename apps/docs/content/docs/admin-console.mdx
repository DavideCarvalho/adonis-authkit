title: Admin Console
description: The opt-in IdP admin console — users, roles, clients, audit.
---

AuthKit ships an opt-in **admin console** mounted under `/admin/*`: a dashboard, user/role
management, the OAuth client list, and the audit log. It is **off by default** — existing
hosts don't change behaviour.

## Enabling it

The console must be turned on in **both** places — `defineConfig` (the guard) and
`registerAuthHost` (route mounting happens before the lazy config resolves):

```ts title="config/authkit_server.ts"
defineConfig({
  // ...
  admin: {
    enabled: true,
    roles: ['ADMIN'], // global roles allowed into /admin; default ['ADMIN']
  },
})
```

```ts title="start/routes.ts"
registerAuthHost(router, {
  mountPath: '/oidc',
  admin: true, // mirror admin.enabled here
})
```

| Field | Type | Default | Notes |
| --- | --- | --- | --- |
| `enabled` | `boolean` | `false` | Mounts the `/admin/*` group behind the admin guard. |
| `roles` | `string[]` | `['ADMIN']` | Global roles that grant access to the console. |

## The guard

The `adminGuard` (also exported) protects the group. It requires:

1. `admin.enabled: true` — when the console is **disabled**, the guard responds with **404
   Not Found** (the routes simply don't exist as far as a caller can tell);
2. an active **account session** (otherwise redirect to `/account/login`);
3. the signed-in account having at least one of `admin.roles` in its global roles
   (otherwise redirect to `/account/tokens`, so the existence of `/admin` isn't leaked).

The allowed roles are resolved at runtime from the `authkit.server` config. See
[Security](/docs/security) for how the disabled-console 404 fits the broader hardening.

## Routes

| Route | Action |
| --- | --- |
| `GET /admin` | Dashboard (counts + recent audit events) |
| `GET /admin/users` | List/search users |
| `POST /admin/users` | Create a user (with a password, or invite via password-reset email) |
| `POST /admin/users/:id/roles` | Replace a user's global roles (`AccountStore.setGlobalRoles`) |
| `POST /admin/users/:id/reset-password` | Issue a reset token and email it to the user |
| `POST /admin/users/:id/disable` | Disable the account (blocks login) — needs `AccountStatusCapability` |
| `POST /admin/users/:id/enable` | Re-enable a disabled account |
| `GET /admin/users/:id/sessions` | List the account's active sessions + grants |
| `POST /admin/users/:id/revoke-sessions` | Revoke all of the account's sessions + grants |
| `GET /admin/clients` | List OAuth clients (static config + adapter-stored) |
| `POST /admin/clients` | Create a dynamic client |
| `POST /admin/clients/:id` | Edit a dynamic client (redirect/post-logout URIs, grants, auth method) |
| `POST /admin/clients/:id/regenerate-secret` | Regenerate a confidential client's secret |
| `POST /admin/clients/:id/delete` | Delete a dynamic client |
| `GET /admin/audit` | Browse the audit log (when the sink supports `list`) |

The console reads through the same contracts as the rest of the host kit:
`AccountStore.listAccounts` / `setGlobalRoles` for users, the OIDC adapter + static
`clients` for the client list, and `AuditSink.list` for audit. A write-only audit sink
degrades gracefully — the audit page shows "query not supported".

## Sessions & revocation

Each user row links to **`/admin/users/:id/sessions`**, which lists the account's active
**sessions** (IdP logins) and **grants** (per-client authorizations), with a live count of
access / refresh tokens per grant. A single **"Revoke all sessions and grants"** button
destroys them via `AdminSessionsService`.

Revocation works by destroying the account's `Session` and `Grant` artifacts directly
through the OIDC adapter. Destroying a grant **cascades** to its tokens: oidc-provider
validates every access/refresh token against `Grant.find(token.grantId)` and rejects it
with `invalid_token` when the grant is gone. As a belt-and-braces measure the service also
destroys the access/refresh token rows that reference the revoked grants (when the adapter
can enumerate them). The operation emits a `session.revoked_all` audit event with the
counts in `metadata`.

Like the client list, enumeration relies on the adapter's **optional** `list?()` capability
(the database and Redis adapters both implement it). When the adapter can't enumerate, the
page **degrades gracefully** with a notice instead of erroring. Both the database adapter's
table query and the Redis adapter's `SCAN` are scoped per model, so the lookup is cheap.

## User management (create / invite / reset / disable)

The users page has a **create** form (email + optional name + optional password). When you
leave the password blank the account is created with a throwaway password and a
**password-reset email** is sent — the user sets their own password (the "invite" flow).
Each row also exposes **Send password reset** and, when the store supports it,
**Disable** / **Enable** buttons. The operations emit `user.created`,
`user.password_reset_sent`, `user.disabled` and `user.enabled` audit events.

Disabling requires the optional **`AccountStatusCapability`** on the account store (see
[Account store](/docs/account-store)). The Lucid store implements it via a
`disabled_at` timestamp column **only when the column exists on your model** — otherwise the
capability is genuinely absent and the disable/enable buttons are hidden. Add the column with
a migration of your own:

```ts
table.timestamp('disabled_at').nullable()
```

Both login paths (the OIDC interaction and the account-console login) reject disabled
accounts with an "account disabled" message, even with a correct password.

## Client CRUD

The clients page lists **both** the statically-configured `clients` (read-only) and the
**dynamically-registered** ones stored in the OIDC adapter (see
[Dynamic Registration](/docs/dynamic-registration)). Adapter-stored clients are fully
manageable from the console via `AdminClientsService`:

- **Create** — generates a `client_id` when omitted; for confidential clients
  (`tokenEndpointAuthMethod !== 'none'`) it generates a `client_secret` and shows it **once**
  (it isn't recoverable afterwards). Public clients get no secret.
- **Edit** — updates the editable metadata (redirect URIs, post-logout redirect URIs,
  grants, token-endpoint auth method) while **preserving** the existing secret.
- **Regenerate secret** — issues a fresh `client_secret` for a confidential client (shown
  once); rejected for public clients.
- **Delete** — removes the client from the adapter.

Every write builds the payload in the exact snake_case shape the underlying oidc-provider
persists (matching what RFC 7591 dynamic registration writes) and then **evicts the
provider's dynamic-client cache** so the change takes effect immediately. Each operation
emits an audit event: `client.created`, `client.updated`, `client.deleted`.

Enumeration relies on the adapter's **optional** `list?()` capability (the legacy
`listClients?()` is still honored for back-compat and now delegates to `list()`). When the
adapter implements neither, the service's `canList` is `false` and the page **degrades
gracefully** — static clients still render and the create form still works, but the
adapter-stored list is hidden with a notice rather than erroring. Statically-configured
clients are always read-only (they live in your config, not the adapter).
