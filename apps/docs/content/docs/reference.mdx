## Runtime Settings

An optional, capability-probed mechanism for persisting configuration at runtime in the
`auth_settings` table. The table is detected on first access; all operations degrade
gracefully when it is absent (fail-safe).

### `auth_settings` schema

```sql
CREATE TABLE auth_settings (
  key        TEXT PRIMARY KEY,
  value      TEXT NOT NULL,        -- JSON
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_by TEXT                  -- nullable account id of the admin
);
```

Create the table with a migration of your own. AuthKit never creates it automatically —
it is an opt-in capability.

### `RuntimeSettings`

Class from `@dudousxd/adonis-authkit-server` (internal, used by the admin console and
API controllers). Accepts a Lucid `Database` instance and an optional `ttlMs` (default
`15_000` — 15 s). Implements `SettingsCapability`.

| Method | Notes |
| --- | --- |
| `getSetting(key)` | Returns `unknown \| null`. Null = table absent or key not found. Cached with TTL. |
| `setSetting(key, value, updatedBy?)` | Upsert. Invalidates cache. No-op when the table is absent. |
| `deleteSetting(key)` | Delete. Invalidates cache. No-op when the table is absent. |
| `listSettings()` | Returns `SettingRow[]`. No cache (low-frequency). Empty array when the table is absent. |
| `isTablePresent()` | Returns `boolean`. Result is memoised (probed once per instance). |
| `invalidate(key?)` | Manually clear the in-memory cache (all keys when called without argument). |

### `SettingsCapability` and `supportsSettings`

```ts
import { supportsSettings } from '@dudousxd/adonis-authkit-server'

if (supportsSettings(store)) {
  const raw = await store.getSetting('bot_protection')
}
```

`supportsSettings(obj)` is a type guard that returns `true` when `obj` exposes a
`getSetting` function (i.e. implements `SettingsCapability`).

### Known setting keys

| Key | Shape | Notes |
| --- | --- | --- |
| `bot_protection` | `{ enabled: boolean; on?: ('login' \| 'signup' \| 'reset')[] }` | Runtime toggle for bot protection. See [Security — Runtime toggle](/docs/security#runtime-toggle). |
| `registration` | `{ enabled: boolean }` | Open or close public sign-up at runtime. Fallback: `config.registration.enabled` (default `true`). See [Security — Registration toggle](/docs/security#registration-toggle). |
| `require_verified_email` | `{ enabled: boolean }` | Override `login.requireVerifiedEmail` for all login flows at runtime. Capability-probed. See [Security — Require verified email toggle](/docs/security#require-verified-email-toggle). |
| `maintenance_mode` | `{ enabled: boolean; message?: string }` | Block login/sign-up/forgot for non-admin users. Admin accounts and the Admin API bypass the gate. See [Security — Maintenance mode](/docs/security#maintenance-mode). |

## Ace commands

Commands provided by `@dudousxd/adonis-authkit-server`:

| Command | Notes |
| --- | --- |
| `node ace authkit:doctor` | Validate the config and peer dependencies; emits findings with `ok`/`warn`/`error` levels. Includes a `checkSettings` probe (see below). Warns when static `clients` are present in config (deprecated). |
| `node ace authkit:eject` | Copy the bundled host-kit controllers/pages into your project for customization. |
| `node ace authkit:users:import --file <path> [--dry-run]` | Bulk-import users from a JSON array or NDJSON file, preserving hashes as-is for lazy rehash on login. |
| `node ace authkit:keys:rotate [--keep=N] [--retire] [--dry-run]` | Rotate signing keys in a managed keystore file; keeps a grace-period window of N previous keys (default 2). |
| `node ace authkit:clients:import [--dry-run]` | Copy the static `clients` from `config/authkit.ts` into the adapter/DB. Idempotent — existing clients (matched by `client_id`) are skipped. The original `client_secret` is preserved for confidential clients. After confirming the import, remove the `clients` block from config. See [below](#authkitclientsimport). |

### `authkit:doctor` — `checkSettings`

When the doctor command probes the `auth_settings` table:

- **Table absent** — silently ok (the feature is opt-in, nothing to warn about).
- **Table present + `botProtection.verify` present in config** — `ok`: runtime settings and
  the bot-protection toggle are active.
- **Table present + `botProtection.verify` absent from config** — `warn`: the
  `bot_protection` row in `auth_settings` is an orphan and has no effect. Either configure
  `botProtection.verify` in `config/authkit.ts` or drop the row.

The doctor also emits a `warn` when `static clients` are present in `config.clients`
(deprecated), reminding you to migrate them with `node ace authkit:clients:import`.
Additionally, it emits a `warn` when `maintenance_mode` or `registration` settings are
stored but the `auth_settings` table is absent, or when `require_verified_email: true` is
set but the `accountStore` lacks the `isEmailVerified` capability.

### `authkit:users:import`

See [Passwords & Migration](/docs/passwords-and-migration#user-import-command) for the file format and field list.

### `authkit:keys:rotate`

See [Security — Signing key rotation](/docs/security#signing-key-rotation) for the flag reference and rotation strategy.

### `authkit:clients:import`

Migrates the static `clients` list from `config/authkit.ts` into the adapter/DB so they can
be managed at runtime via the admin console or Admin REST API.

```bash
# Preview what would be created (no writes)
node ace authkit:clients:import --dry-run

# Run the migration
node ace authkit:clients:import
```

**Behaviour:**

- Each entry in `config.clients` is attempted in order.
- If a client with the same `client_id` **already exists** in the adapter, the entry is
  **skipped** (idempotent — the existing secret is preserved).
- Confidential clients (`clientSecret` set) have their original secret preserved during
  import. The secret is printed **once** for each newly created client — save it before the
  command exits.
- Public clients (no `clientSecret`) are imported without a secret.
- The command prints a per-client `created` / `skipped` / `error` report and exits non-zero
  if any entry failed.

**After the import:**

1. Verify the clients appear in the admin console (`/admin/clients`) or Admin API.
2. Remove the `clients: [...]` block from `config/authkit.ts` (or leave it empty: `clients: []`).
3. Re-run `node ace authkit:doctor` — the deprecation warning should be gone.

**Requirement:** the adapter must be persistent (e.g. `adapters.database(...)`). If a
volatile adapter (Redis without AOF/RDB, in-memory) is used, `authkit:doctor` will warn
that clients managed at runtime will be lost on restart.
