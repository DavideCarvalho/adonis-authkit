description: Read the user in your controllers with ctx.auth — login, identity vs user, guards, and logout.

This is the practical guide for a relying party (RP): once the client kit is wired, every
request carries an authenticator on `ctx.auth`. Here you'll log a user in, read them in a
controller, gate routes by role, and log them out. For the conceptual model — resolvers,
the `Identity` shape, `resolveUser` — see [Client](/docs/client).

## Setup recap

Two pieces make `ctx.auth` exist. First, `config/authkit_client.ts` (published by
`node ace configure @dudousxd/adonis-authkit-client`) declares the issuer, client
credentials, the resolver, and how to map an identity to your user:

```ts title="config/authkit_client.ts"
import env from '#start/env'
import { defineConfig, resolvers } from '@dudousxd/adonis-authkit-client'
import type { Identity } from '@dudousxd/adonis-authkit-client'
import AppUser from '#models/app_user'

const authkitClientConfig = defineConfig({
  issuer: env.get('AUTHKIT_ISSUER'),
  clientId: env.get('AUTHKIT_CLIENT_ID'),
  clientSecret: env.get('AUTHKIT_CLIENT_SECRET'),
  redirectUri: env.get('AUTHKIT_REDIRECT_URI'),
  resolver: resolvers.jwt({ tokenSource: 'session' }),

  // Map the IdP identity (claims) to YOUR user record.
  resolveUser: async (identity: Identity) => {
    return AppUser.updateOrCreate(
      { id: identity.userId },
      { id: identity.userId, email: identity.email, fullName: identity.profile?.name ?? null }
    )
  },
})

export default authkitClientConfig
```

Second, `authkit_middleware`. Register it as a **router-global** middleware so every route
gets `ctx.auth`:

```ts title="start/kernel.ts"
import router from '@adonisjs/core/services/router'

router.use([
  // ...existing middleware
  () => import('@dudousxd/adonis-authkit-client/authkit_middleware'),
])
```

The middleware does two things per request — it proactively refreshes the token set
(`maybeRefresh`, see [Refresh Tokens](/docs/refresh-tokens)) and then attaches the
authenticator:

```ts title="authkit_middleware.ts (excerpt)"
const manager = await ctx.containerResolver.make('authkit.client')
await manager.maybeRefresh(ctx)
ctx.auth = await manager.createAuthenticator(ctx)
return next()
```

<Callout type="info">
  `ctx.auth` is **always present** after the middleware runs — it never throws on an
  anonymous request. Whether someone is actually logged in is decided by `getIdentity()`
  resolving to a non-`null` value. Use `authenticate()` / `check()` for that (below).
</Callout>

You can also register it as a **named** middleware if you'd rather opt routes in
explicitly instead of globally:

```ts title="start/kernel.ts"
export const middleware = router.named({
  // ...
  authkit: () => import('@dudousxd/adonis-authkit-client/authkit_middleware'),
})
```

```ts title="start/routes.ts"
router.get('/dashboard', [DashboardController]).use(middleware.authkit())
```

## How login works

Login is OIDC authorization-code + PKCE. The bundled `OidcSessionController` stub
(published on configure) is the whole flow — three short methods over the client helpers:

```ts title="app/controllers/oidc_session_controller.ts"
import type { HttpContext } from '@adonisjs/core/http'
import { buildAuthorizeUrl, exchangeCode, generatePkce } from '@dudousxd/adonis-authkit-client'
import { randomUUID } from 'node:crypto'

export default class OidcSessionController {
  async login(ctx: HttpContext) {
    const manager = await ctx.containerResolver.make('authkit.client')
    const cfg = manager.clientConfig
    const { verifier, challenge } = await generatePkce()
    const state = randomUUID()
    // Stash PKCE verifier + state so we can verify the callback.
    ctx.session.put('authkit_pkce', { verifier, state })
    const url = buildAuthorizeUrl({
      issuer: cfg.issuer, clientId: cfg.clientId, redirectUri: cfg.redirectUri,
      scopes: cfg.scopes, state, codeChallenge: challenge,
    })
    return ctx.response.redirect(url)
  }

  async callback(ctx: HttpContext) {
    const manager = await ctx.containerResolver.make('authkit.client')
    const cfg = manager.clientConfig
    const { code, state } = ctx.request.qs()
    const pkce = ctx.session.get('authkit_pkce')
    if (!pkce || pkce.state !== state) return ctx.response.badRequest({ error: 'state inválido' })
    const tokenSet = await exchangeCode({
      issuer: cfg.issuer, clientId: cfg.clientId, clientSecret: cfg.clientSecret,
      redirectUri: cfg.redirectUri, code, codeVerifier: pkce.verifier,
    })
    // Persist the token set under the configured sessionKey — this is what
    // resolvers.jwt({ tokenSource: 'session' }) reads on later requests.
    ctx.session.put(cfg.sessionKey, tokenSet)
    ctx.session.forget('authkit_pkce')
    return ctx.response.redirect('/')
  }

  async logout(ctx: HttpContext) {
    const manager = await ctx.containerResolver.make('authkit.client')
    ctx.session.forget(manager.clientConfig.sessionKey)
    return ctx.response.redirect('/')
  }
}
```

Wire the routes (these are the `redirectUri` you registered for the client):

```ts title="start/routes.ts"
const OidcSessionController = () => import('#controllers/oidc_session_controller')

router.get('/auth/login', [OidcSessionController, 'login']).as('auth.login')
router.get('/auth/callback', [OidcSessionController, 'callback']).as('auth.callback')
router.post('/auth/logout', [OidcSessionController, 'logout']).as('auth.logout')
```

The sequence:

1. `login` → `buildAuthorizeUrl` redirects the browser to the IdP's `/authorize`.
2. The user authenticates at the IdP, which redirects back to `redirectUri` with `code` +
   `state`.
3. `callback` verifies `state`, calls `exchangeCode` to swap the code for a token set, and
   stores it with `session.put(cfg.sessionKey, tokenSet)`.

That's it — there's nothing else to "log in" on the RP. The stored token set is the login.

## How the user lands on the request

On every subsequent request the middleware builds `ctx.auth`, and the resolver reads the
token set from the session. Two distinct things come out of it:

- **Identity** — the OIDC claims, app-agnostic. Same shape for every resolver:

  ```ts
  interface Identity {
    userId: string          // sub
    email: string
    globalRoles: string[]   // from the configured claim, default 'roles'
    profile?: { name?: string; avatarUrl?: string }
    sessionId?: string      // sid
    issuedAt: number; expiresAt: number
    raw: Record<string, unknown>
  }
  ```

- **User** — *your* domain model, produced by `resolveUser(identity)`. The Identity says
  *who* the IdP authenticated; `resolveUser` turns that into the row your app works with.
  A typical mapping keys on `userId` (the `sub`), falling back to `email` if you let users
  exist before their first OIDC login:

  ```ts title="config/authkit_client.ts"
  resolveUser: async (identity: Identity) => {
    // Prefer the stable subject; reconcile a pre-existing row by email on first login.
    let user = await AppUser.findBy('id', identity.userId)
    if (!user) user = await AppUser.findBy('email', identity.email)
    if (!user) user = new AppUser().fill({ id: identity.userId })
    user.merge({ id: identity.userId, email: identity.email })
    await user.save()
    return user
  }
  ```

`resolveUser` runs lazily, only when you call `auth.getUser()` — see below.

## Controller usage

This is the part you'll use every day. `ctx.auth` is an `Authenticator`; destructure it
from the context.

### Require a login — `authenticate()`

`authenticate()` resolves the identity and **throws** if there isn't one. Use it on
protected endpoints where anonymous access is a programming error:

```ts title="app/controllers/dashboard_controller.ts"
import type { HttpContext } from '@adonisjs/core/http'

export default class DashboardController {
  async handle({ auth, inertia }: HttpContext) {
    const identity = await auth.authenticate() // throws if not logged in
    return inertia.render('dashboard', { email: identity.email })
  }
}
```

### Get your user — `getUser()`

`getUser()` returns whatever `resolveUser` produced (or `null` if not authenticated). It's
the usual entry point because you almost always want your model, not raw claims:

```ts
async handle({ auth, response }: HttpContext) {
  const user = await auth.getUser<AppUser>()
  if (!user) return response.unauthorized({ message: 'Não autenticado' })
  return user
}
```

<Callout type="info">
  `getUser()` returns `unknown` (the kit doesn't know your model). Pass a type argument —
  `auth.getUser<AppUser>()` — or assert the result to get full typing.
</Callout>

### Optional auth — `check()`

`check()` returns a boolean and never throws. Use it for pages that render differently for
guests vs. members:

```ts
async handle({ auth, inertia }: HttpContext) {
  if (await auth.check()) {
    const user = await auth.getUser<AppUser>()
    return inertia.render('home', { user })
  }
  return inertia.render('home', { user: null })
}
```

### Raw claims — `getIdentity()`

When you need the claims directly (the `sub`, `sessionId`, anything in `raw`) without
hitting your DB, use `getIdentity()` — it returns `Identity | null`:

```ts
const identity = await auth.getIdentity()
if (identity) console.log(identity.userId, identity.sessionId, identity.raw)
```

### Roles — `hasGlobalRole` / `hasAppRole`

Two role surfaces, mirroring Identity vs User:

- **`hasGlobalRole(role)`** is synchronous and checks `identity.globalRoles` (claims from
  the IdP). It only sees roles after the identity has been resolved — call `authenticate()`
  / `check()` / `getIdentity()` first.
- **`hasAppRole(role)`** is `async` and consults `resolveAppRoles(identity)` — your
  app-local roles. It resolves the identity for you.

```ts
async handle({ auth, response }: HttpContext) {
  await auth.authenticate()                 // ensure identity is resolved
  if (!auth.hasGlobalRole('admin')) {       // sync, from claims
    return response.forbidden({ message: 'Acesso negado' })
  }
  if (await auth.hasAppRole('billing.manager')) {
    // app-local capability
  }
}
```

### Protecting a route group

Centralize the check in a small middleware (or the named `authkit` middleware from setup)
and apply it to a group:

```ts title="start/routes.ts"
router
  .group(() => {
    router.get('/dashboard', [DashboardController]).as('dashboard')
    router.get('/settings', [SettingsController]).as('settings')
  })
  .prefix('/app')
  .as('app')
  .use(middleware.requireAuth()) // a named middleware that calls auth.authenticate()
```

### Redirect-to-login instead of throwing

For browser routes you usually want a redirect, not a 500. Use `check()` and bounce to the
login route:

```ts title="app/middleware/require_auth_middleware.ts"
import type { HttpContext } from '@adonisjs/core/http'
import type { NextFn } from '@adonisjs/core/types/http'

export default class RequireAuthMiddleware {
  async handle(ctx: HttpContext, next: NextFn) {
    if (!(await ctx.auth.check())) {
      return ctx.response.redirect().toRoute('auth.login')
    }
    return next()
  }
}
```

### Typing `ctx.auth`

You don't have to declare anything. The client package ships a module augmentation
(`interface HttpContext { auth: Authenticator }`) registered via its `adonisjs.types`
entry, so once the package is configured, `ctx.auth` is typed everywhere with no import:

```ts
async handle({ auth }: HttpContext) {
  await auth.authenticate() // fully typed
}
```

## Logout

Logout has two parts. The minimal stub just drops the local session (`session.forget`).
For a full **RP-initiated** logout that also ends the session at the IdP, redirect to the
end-session endpoint with the stored `id_token` as the hint, then clear the local session:

```ts title="app/controllers/oidc_session_controller.ts"
import { buildEndSessionUrl } from '@dudousxd/adonis-authkit-client'

async logout(ctx: HttpContext) {
  const manager = await ctx.containerResolver.make('authkit.client')
  const cfg = manager.clientConfig
  // id_token_hint lets the IdP skip its confirmation page.
  const idToken = manager.getIdToken(ctx)
  ctx.session.forget(cfg.sessionKey)
  return ctx.response.redirect(
    buildEndSessionUrl({
      issuer: cfg.issuer,
      idToken,
      clientId: cfg.clientId,
      postLogoutRedirectUri: 'https://my-app.example.com/',
    })
  )
}
```

`buildEndSessionUrl` targets `${issuer}/session/end`. The `postLogoutRedirectUri` must be
registered on the client. See [Security](/docs/security#logout) for the details and
[Back-Channel Logout](/docs/back-channel-logout) for IdP-initiated termination.
